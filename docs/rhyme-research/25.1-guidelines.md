# Version 25.1 Guidelines - Fast Rhyme Readiness Pass

Research date: 2026-07-04

## Goal

Version 25.1 should begin fixing the visible delay before rhyme suggestions appear. The app works, but the current rhyme row still waits for the legacy engine to finish loading large text assets before it can return results. This update should make the fast rhyme row available as early as possible without changing rhyme quality, storage, playback, recording, settings, appcast behavior, or user data.

Do not make Rhyme Engine V2 the default scorer in 25.1. Use this release to improve readiness, asset loading, instrumentation, and cache delivery first.

## Current Code Findings

The visible suggestions still come from the legacy `RhymeEngine`, not from `rhyme_lexicon_v2.tflex`.

Relevant files:

- `app/src/main/java/com/davehq/thetopflow/RhymeEngine.java`
- `app/src/main/kotlin/com/davehq/thetopflow/NotesViewModel.kt`
- `app/src/main/kotlin/com/davehq/thetopflow/rhyme/RhymeAssetStore.kt`
- `app/build.gradle`

Current runtime path:

1. `NotesViewModel` creates `RhymeEngine(application)`.
2. `NotesViewModel.init` calls `rhymeEngine.loadAsync { ... }`.
3. `RhymeEngine.loadAsync()` starts a background thread.
4. That thread calls `load()`.
5. `load()` reads the prepared index, pronunciation overrides, common words, fast hot cache, and expanded hot cache.
6. Only after all of that does `ready = true`.
7. `NotesViewModel.scheduleRhymeUpdate()` refuses to ask for suggestions while `!rhymeEngine.isReady()`.
8. The UI shows `Rhyme engine loading` until `rhymeLoading=false`.

The critical delay comes from the all-or-nothing `ready` gate.

## Current Asset Load Order

`RhymeEngine.load()` currently does this:

1. Clear all maps and caches.
2. Try `loadPreparedIndex()`.
3. If that fails, load `cmudict.dict`.
4. Add pronunciation overrides.
5. Add common slang/common rhyme words.
6. Load `rhyme_hot_cache.tfcache`.
7. Load `rhyme_expanded_hot_cache.tfcache`.

This order is bad for perceived speed because the app reads and parses the full prepared index before the fast row cache can be used.

Current asset sizes observed locally:

- `rhyme_index_accel.tfindex`: about 7.2 MB
- `rhyme_index.tsv`: about 7.2 MB
- `rhyme_hot_cache.tfcache`: about 1.3 MB
- `rhyme_expanded_hot_cache.tfcache`: about 1.5 MB
- `rhyme_lexicon_v2.tflex`: about 6.3 MB
- `cmudict.dict`: about 3.6 MB

The current fast hot cache is useful, but it is loaded too late and parsed as text with `BufferedReader`, `line.split()`, string arrays, and map insertion. That creates unnecessary startup allocation.

## Web Research Summary

Android `AssetManager.openFd()` opens an uncompressed asset by memory mapping it and returns an `AssetFileDescriptor`. The asset must be uncompressed, or an exception is thrown. This supports the 25.0 direction of keeping binary `.tfcache`, `.tfindex`, and `.tflex` assets uncompressed and reading them through mmap.

Source: https://developer.android.com/reference/android/content/res/AssetManager#openFd(java.lang.String)

Android startup guidance says resource-heavy operations should be measured and moved off the critical startup path when possible. Non-essential resources can wait until after startup completes. For The Top Flow, expanded rhyme search and full index parsing are not required before the fast row can show cached suggestions.

Source: https://developer.android.com/topic/performance/appstartup/analysis-optimization

Android `StrictMode` is commonly used to catch accidental disk or network access on the main thread. It is useful here as a debug-only guard to prove no rhyme asset loading or parsing touches the UI thread.

Source: https://developer.android.com/reference/kotlin/android/os/StrictMode

Baseline Profiles improve code execution speed by avoiding interpretation and just-in-time compilation for included code paths. Startup Profiles can additionally improve app startup by improving DEX layout. The app already has profile infrastructure, so 25.1 should include the fast rhyme path in the generated profile route if practical.

Sources:

- https://developer.android.com/topic/performance/baselineprofiles/overview
- https://developer.android.com/topic/performance/startupprofiles/dex-layout-optimizations

Kotlin Flow `debounce` emits only after newer values stop arriving within the timeout. The current manual `140ms` delay in `scheduleRhymeUpdate()` is therefore expected to add visible latency after typing. Once a fast cache is ready, this timeout should be shorter.

Source: https://kotlinlang.org/api/kotlinx.coroutines/kotlinx-coroutines-core/kotlinx.coroutines.flow/debounce.html

## 25.1 Implementation Strategy

25.1 should have one job: make the first fast-row rhyme suggestions appear before the full engine is ready.

Implement this in phases:

1. Add phased readiness to `RhymeEngine`.
2. Load the fast hot cache first.
3. Publish fast readiness immediately after the fast cache is usable.
4. Continue loading the full prepared index and expanded cache in the background.
5. Let `NotesViewModel` use cached fast suggestions when fast-ready, even if full-ready is false.
6. Add timing instrumentation.
7. Keep all existing fallback paths.

## New Readiness Model

Replace the single effective readiness concept with two readiness states:

- `fastReady`: the fast hot cache can answer default fast-row suggestions.
- `ready`: the full legacy engine is loaded and existing scorer/fallback behavior is available.

Keep the existing `isReady()` method returning full readiness to minimize behavior changes. Add a new method:

```java
boolean isFastReady() {
    return fastReady;
}
```

Add fields:

```java
private volatile boolean fastReady = false;
private volatile boolean ready = false;
private volatile boolean loading = false;
```

Add optional stage timings:

```java
private volatile long fastCacheLoadMs = -1L;
private volatile long preparedIndexLoadMs = -1L;
private volatile long expandedCacheLoadMs = -1L;
private volatile long fullLoadMs = -1L;
```

Add a callback model that can publish both stages:

```java
interface LoadCallbacks {
    void onFastReady();
    void onFullReady();
}
```

Keep the old API for compatibility:

```java
void loadAsync(Runnable onReady) {
    loadAsync(new LoadCallbacks() {
        @Override public void onFastReady() {}
        @Override public void onFullReady() {
            if (onReady != null) onReady.run();
        }
    });
}
```

Add:

```java
void loadAsync(LoadCallbacks callbacks) {
    if (ready || loading) return;
    loading = true;
    new Thread(() -> {
        long start = System.currentTimeMillis();
        loadFastCacheFirst();
        if (fastReady && callbacks != null) callbacks.onFastReady();
        loadFullEngineAfterFastCache();
        ready = true;
        loading = false;
        fullLoadMs = System.currentTimeMillis() - start;
        clearCache();
        Log.d(TAG, "rhyme_trace stage=full_ready words=" + phonesByWord.size()
                + " exactKeys=" + exactIndex.size()
                + " fastHotCache=" + fastHotCache.size()
                + " expandedHotCache=" + expandedHotCache.size()
                + " fastCacheMs=" + fastCacheLoadMs
                + " preparedIndexMs=" + preparedIndexLoadMs
                + " expandedCacheMs=" + expandedCacheLoadMs
                + " fullMs=" + fullLoadMs);
        if (callbacks != null) callbacks.onFullReady();
    }, "TopFlowRhymeLoad").start();
}
```

Do not call `clearCache()` after publishing `fastReady` unless it is required. Clearing result cache after full load is acceptable but be careful not to wipe a useful fast result and cause a visible empty row. If `clearCache()` must stay after full load, immediately refresh selected rhymes from `NotesViewModel` on full-ready.

## Load Order Change

Create two methods:

```java
private void loadFastCacheFirst() {
    long start = System.currentTimeMillis();
    fastHotCache.clear();
    fastHotCacheReady = loadHotCache(
            FAST_HOT_CACHE_ASSET,
            FAST_HOT_CACHE_HEADER,
            FAST_HOT_CACHE_MAX_CANDIDATES,
            FAST_HOT_CACHE_LIMIT,
            fastHotCache
    );
    fastReady = fastHotCacheReady && !fastHotCache.isEmpty();
    fastCacheLoadMs = System.currentTimeMillis() - start;
    Log.d(TAG, "rhyme_trace stage=fast_ready ready=" + fastReady
            + " rows=" + fastHotCache.size()
            + " ms=" + fastCacheLoadMs);
}
```

Then:

```java
private void loadFullEngineAfterFastCache() {
    phonesByWord.clear();
    exactIndex.clear();
    familyIndex.clear();
    dictionaryWords.clear();

    long indexStart = System.currentTimeMillis();
    if (!loadPreparedIndex()) loadCmuDictionary();
    preparedIndexLoadMs = System.currentTimeMillis() - indexStart;

    addPronunciationOverrides();
    for (String word : COMMON_RHYME_WORDS) {
        String w = normalizeWord(word);
        String phones = slangPhones(w, true);
        if (!w.isEmpty() && !phones.isEmpty() && !dictionaryWords.contains(w)) {
            addEntry(w, phones, true);
        }
    }

    long expandedStart = System.currentTimeMillis();
    expandedHotCacheReady = loadHotCache(
            EXPANDED_HOT_CACHE_ASSET,
            EXPANDED_HOT_CACHE_HEADER,
            EXPANDED_HOT_CACHE_MAX_CANDIDATES,
            EXPANDED_HOT_CACHE_LIMIT,
            expandedHotCache
    );
    expandedCacheLoadMs = System.currentTimeMillis() - expandedStart;
}
```

Important: do not clear `fastHotCache` in `loadFullEngineAfterFastCache()`. It must stay available.

## Fast Suggestion API

The current `suggest()` returns empty while `!ready`. That prevents fast cache use before full load.

Change the first guard:

```java
if (base.isEmpty()) return out;
if (!ready && !fastReady) return out;
```

Then attempt hot cache before full readiness:

```java
ArrayList<String> hot = hotCacheSuggestion(base, limit, maxCandidates, options);
if (hot != null) {
    cache result;
    return hot;
}
if (!ready) return out;
```

This lets fast cache hits work before the full index exists. Cache misses still return empty until full readiness, preserving safety.

Add trace logging around suggestion latency:

```java
long start = System.nanoTime();
String source = "empty";
...
source = "hot_cache";
...
source = "full_scorer";
...
Log.d(TAG, "rhyme_trace stage=suggest word=" + base
        + " source=" + source
        + " ready=" + ready
        + " fastReady=" + fastReady
        + " limit=" + limit
        + " maxCandidates=" + maxCandidates
        + " count=" + out.size()
        + " ms=" + ((System.nanoTime() - start) / 1_000_000.0));
```

Keep logs lightweight. If logs become noisy, gate per distinct word or debug builds.

## NotesViewModel Changes

Update `NotesViewModel.init` to receive both readiness callbacks.

Recommended structure:

```kotlin
rhymeEngine.loadAsync(object : RhymeEngine.LoadCallbacks {
    override fun onFastReady() {
        viewModelScope.launch {
            rhymeLoading.value = false
            refreshRhymesForSelected()
        }
    }

    override fun onFullReady() {
        viewModelScope.launch {
            rhymeLoading.value = false
            refreshRhymesForSelected()
        }
    }
})
```

If `RhymeEngine.LoadCallbacks` cannot be public due package-private Java/Kotlin access, keep it package-private in `com.davehq.thetopflow`; `NotesViewModel` is in the same package.

Update loading checks in `scheduleRhymeUpdate()`:

Current:

```kotlin
if (!rhymeEngine.isReady()) {
    rhymeLoading.value = true
    return@launch
}
```

Recommended:

```kotlin
if (!rhymeEngine.isFastReady() && !rhymeEngine.isReady()) {
    rhymeLoading.value = true
    return@launch
}
```

After suggestions:

```kotlin
rhymeLoading.value = !rhymeEngine.isFastReady() && !rhymeEngine.isReady()
```

## Rhyme Update Delay Change

Current delay is `140ms`.

Keep `140ms` while engine is not fast-ready. Once fast-ready, use shorter delay:

```kotlin
private fun scheduleRhymeUpdate(body: String, delayMs: Long = defaultRhymeDelayMs()) {
    ...
}

private fun defaultRhymeDelayMs(): Long {
    return if (rhymeEngine.isFastReady()) 50L else 140L
}
```

Use `delayMs = 0` for:

- note open
- fast-ready callback
- full-ready callback

This means:

- startup is protected while loading
- typing still has debounce
- cached fast suggestions appear sooner once the cache is ready

## Binary Fast Cache For 25.1

If time allows, the biggest structural improvement is replacing text hot cache parsing with a binary mmap asset. This is the preferred 25.1 implementation if the coder can complete it safely.

Add a new asset:

```text
app/src/main/assets/rhyme_fast_cache_v2.tfcache
```

Keep old `rhyme_hot_cache.tfcache` as fallback for one release.

Suggested binary format:

Header:

```text
magic: 8 bytes, "TFCACH2\0" or "TFCACHE2"
version: uint16, 2
rowCount: uint32
rowTableOffset: uint32
stringTableOffset: uint32
checksum: uint32
reserved: uint32
```

Row:

```text
wordOffset: uint32
suggestionCount: uint16
suggestion1Offset: uint32
suggestion2Offset: uint32
suggestion3Offset: uint32
suggestion4Offset: uint32
suggestion5Offset: uint32
suggestion6Offset: uint32
```

All string offsets should be absolute offsets in the file, not relative offsets. This avoids the bug fixed in 25.0 for `rhyme_lexicon_v2.tflex`.

Sort rows by normalized word. Runtime lookup should use binary search over the mapped row table.

Add builder:

```text
tools/build_rhyme_fast_cache_v2.py
```

Builder should:

1. Read current `rhyme_hot_cache.tfcache`.
2. Validate header contains:
   - `strictness=Balanced`
   - `exactOnly=false`
   - `includeSlang=true`
   - `maxCandidates=360`
   - `limit=6`
3. Parse rows.
4. Normalize the word.
5. Keep first 6 suggestions.
6. Drop rows with fewer than 4 suggestions.
7. Sort by normalized word.
8. Emit binary asset.
9. Emit debug TSV:
   - `build/rhyme-v2-debug/fast_cache_debug.tsv`

Add validator:

```text
tools/validate_rhyme_fast_cache_v2.py
```

Validator should check:

1. Asset exists.
2. Magic matches.
3. Version matches.
4. Checksum matches.
5. Row count is greater than or equal to current text hot cache row count minus known filtered rows.
6. Required words exist:
   - `my`
   - `try`
   - `time`
   - `out`
   - `running`
   - `moving`
   - `runnin`
   - `movin`
7. Each required word has at least 4 suggestions.
8. String offsets are in range.
9. Row table does not run past `stringTableOffset`.
10. Rows are sorted by normalized word.

Runtime class idea:

```java
final class RhymeFastCacheStore {
    private ByteBuffer buffer;
    private int rowCount;
    private int rowTableOffset;
    private int stringTableOffset;

    boolean load(Context context) { ... }
    String[] lookup(String normalizedWord, int limit) { ... }
}
```

Use `RhymeAssetStore.openMappedAsset(context, "rhyme_fast_cache_v2.tfcache")`.

Because `RhymeAssetStore` is Kotlin in package `com.davehq.thetopflow.rhyme`, either:

1. Move/duplicate asset-mapping helper into Java package-accessible code, or
2. Expose `RhymeAssetStore` cleanly with `@JvmStatic`, or
3. Implement minimal Java mmap helper in `RhymeEngine.java`.

Recommended: add `@JvmStatic` to `RhymeAssetStore.openMappedAsset` so Java can call:

```java
ByteBuffer mapped = RhymeAssetStore.openMappedAsset(context, "rhyme_fast_cache_v2.tfcache");
```

If package import friction is high, do not block 25.1 on this. Load text hot cache first and defer binary cache to 25.2.

## Fallback Rules

Fallback must be conservative:

1. Try binary fast cache first.
2. If binary asset is missing, corrupt, compressed, or unsupported, log `rhyme_trace stage=fast_cache_binary_failed reason=...`.
3. Fall back to current text `rhyme_hot_cache.tfcache`.
4. If text hot cache fails, keep old behavior: wait for full engine.
5. Never crash the app because of a rhyme cache asset.

## Macrobenchmark Requirements

Add or extend a Macrobenchmark route for:

1. Cold launch to notes grid visible.
2. Open latest note.
3. Wait for `rhyme_suggestions` node.
4. Type a short line ending in a common cached word, such as `time`.
5. Measure time until suggestion row contains at least one expected suggestion.

Suggested expected words:

- Query `time`: expect one of `rhyme`, `dime`, `climb`, `lime`.
- Query `out`: expect one of `bout`, `clout`, `shout`, `doubt`, `about`.
- Query `running`: expect `runnin`.

Performance target for 25.1:

- Cached fast suggestion after `fastReady`: P95 <= 50 ms from scheduled query execution.
- First visible cached rhyme after editor opens: target <= 500 ms on Pixel after process start, then tighten in later versions.
- Full engine can still take longer, as long as fast cache is available first.

## Acceptance Criteria

25.1 is accepted when:

- Fast cached rhyme suggestions can appear before full engine readiness.
- Existing full scorer behavior remains available after full load.
- Missing or corrupt binary cache falls back safely.
- Existing rhyme quality check passes.
- Existing V2 asset validation passes.
- Release build succeeds.
- App update manifest points to the GitHub Release asset.
- GitHub Release `v25.1` exists with the APK attached.
- Notes, playback, recording, settings, saved data, and update checking remain unaffected.
