# 25.5 To 32.0 Rhyme Roadmap

Research date: 2026-07-04

Goal: make The Top Flow's rhyme section feel instant, stay local/offline, sit near the active writing line, and improve hip-hop rhyme quality through small releasable updates from 25.5 through 32.0.

## Research Findings

- Android `AssetManager.openFd()` can memory-map uncompressed assets and throws if an asset is compressed. Keep rhyme binary assets in `noCompress` and prefer mmap reads for lexicon, cache, candidate, and phrase data. Source: https://developer.android.com/reference/android/content/res/AssetManager
- Android Baseline Profiles can improve startup and runtime code paths from first launch by precompiling critical interactions. Add rhyme editor open, active-line typing, and suggestion refresh paths to profile generation. Source: https://developer.android.com/topic/performance/baselineprofiles/overview
- SQLite FTS5 provides persistent indexed search and prefix-query support, but direct rhyme lookup should remain purpose-built binary candidate tables because rhyme queries are phonetic/bucketed, not natural text search. FTS5 is useful later for optional word/reference browsing. Source: https://www.sqlite.org/fts5.html
- Hirjee and Brown's ISMIR 2009 rap-rhyme work supports scoring imperfect and internal rhymes by phoneme similarity rather than only exact rule matching. This supports V2 coda/vowel/stress scoring and internal line analysis. Source: https://ismir2009.ismir.net/proceedings/OS8-1.pdf
- CMUdict already gives a strong local base: open machine-readable pronunciations, ARPAbet phones, and vowel stress markers. The app should keep CMUdict as a source but move runtime use into generated binary assets. Source: https://www.speech.cs.cmu.edu/cgi-bin/cmudict
- Datamuse/RhymeZone product behavior is useful as a benchmark: constraints, sound matching, metadata, context hints, autocomplete, similar-sounding fallback, spelling correction, and related/contextual suggestions. The app should replicate the local shape, not call the API for core suggestions. Sources: https://www.datamuse.com/api/ and https://datamuse.com/api/users.html

## Release Track

### 25.5 - Active-Line Rhyme Dock Foundation

Implement a text selection aware editor body and place the compact rhyme strip directly below the current active line when possible. Keep the existing row fallback if the line cannot be measured.

Acceptance:
- Typing a word shows rhyme chips near the active line, not in a distant panel.
- Existing insert-chip behavior still appends text correctly.
- `assembleRelease` passes and 25.5 is in GitHub Releases/appcast.

### 25.6 - Active-Line Stability Pass

Prevent the dock from covering the keyboard, action row, or long note text. Add line-anchor guards for very tall notes, selection changes, and paste operations.

Acceptance:
- Rhyme dock does not overlap editor controls.
- Cursor movement updates the active line anchor.

### 25.7 - Fast Cache Query Contract

Make fast-row request limits, cache row limits, and logging agree. Add validation that app fast-row calls are eligible for binary cache lookup.

Acceptance:
- A validator proves the app's default `suggest()` call is fast-cache eligible.
- `rhyme_trace` reports real binary-cache row counts.

### 25.8 - Hot Cache Expansion

Increase common hip-hop query coverage by adding more slang, common endings, and phrase tails to `rhyme_fast_cache_v2.tfcache`.

Acceptance:
- Required query list covers at least 100 common writing words.
- Missing cache row rate is logged and reduced.

### 25.9 - Baseline Profile Rhyme Path

Add or refresh baseline profile coverage for app launch, editor open, body typing, and first rhyme suggestion.

Acceptance:
- APK contains generated baseline profile data.
- Macrobenchmark or generation task covers editor/rhyme interactions.

### 26.0 - Binary Candidate Table V2

Generate `rhyme_candidates_v2.tfcand` from CMUdict plus curated slang. Store candidate buckets by word ID so runtime lookup avoids full candidate scans.

Acceptance:
- Candidate asset validates header/checksum/row count.
- Runtime V2 lookup exists behind a disabled flag.

### 26.1 - V2 Quality Golden Set

Add a golden query suite for exact, near, slant, slang, and bad-pair exclusions.

Acceptance:
- V2 quality script passes without changing production default.

### 26.2 - V2 Performance Harness

Add a microbenchmark for V2 lookup and Android timing logs for first visible suggestion.

Acceptance:
- Local benchmark reports P50/P95 lookup times.

### 27.0 - V2 Default For Compact Row

Route the compact row to V2 for eligible queries, with old engine fallback.

Acceptance:
- Compact row uses V2 for common/default options.
- Old engine fallback works if assets fail.

### 27.1 - V2 Ranking Tuning

Tune bucket weights for hip-hop: perfect, near, slant, assonance, consonance, slang variants, and phrase tails.

Acceptance:
- Golden top rows are equal or better than old engine.

### 27.2 - Expanded Rhyme UI Rework

Use V2 buckets in expanded rhyme views and label rhyme types clearly.

Acceptance:
- Expanded results expose perfect/near/slant/multis without slowing typing.

### 28.0 - Active Context Analyzer

Analyze the current note incrementally to prefer rhyme families already used in the line/bar while avoiding repeated obvious words.

Acceptance:
- Suggestions respond to current bar context without scanning the whole note on each keystroke.

### 28.1 - Internal Rhyme Suggestions

Use the active line to suggest internal rhyme placements and mid-line compatible rhymes.

Acceptance:
- Internal rhyme hints are optional and do not replace the compact row.

### 28.2 - Repetition And Cliche Filters

Downrank repeated words and tired exact matches when better hip-hop near/slant choices exist.

Acceptance:
- Common exact rhymes remain available but do not dominate every row.

### 29.0 - Phrase And Multisyllabic Candidate Asset

Generate phrase-tail and multisyllabic candidate tables for 2-6 syllable tails.

Acceptance:
- Phrase suggestions are local, precomputed, and fast.

### 29.1 - Phrase Ranking

Rank multis by tail match, stress match, phrase frequency, and context fit.

Acceptance:
- Phrase suggestions are useful for common line endings.

### 29.2 - Phrase UI Pass

Expose phrase/multis without overwhelming the compact row.

Acceptance:
- Compact row stays quick; expanded panel handles depth.

### 30.0 - OOV And Slang Pronunciation

Add local grapheme/slang fallback for unknown words, starting with rule-based endings before any heavier model.

Acceptance:
- Unknown hip-hop spellings produce reasonable candidates offline.

### 30.1 - Slang Lexicon Growth

Expand curated slang and pronunciation overrides with reviewable source notes.

Acceptance:
- License/source notes remain clean.

### 30.2 - OOV Quality Guard

Add bad-guess tests so OOV does not poison obvious results.

Acceptance:
- OOV improves coverage without degrading golden queries.

### 31.0 - Local Reference Store

Move more lookup/reference data into phone-local binary or SQLite-backed storage for quick browsing and optional detail views.

Acceptance:
- Reference lookup does not block rhyme suggestions.

### 31.1 - Reference Search

Add fast local search over words, phones, syllables, and rhyme families.

Acceptance:
- Search works offline and does not add typing latency.

### 31.2 - Final Performance Compression

Trim duplicate assets, remove obsolete fallback paths where safe, and tighten load order.

Acceptance:
- Startup and first-rhyme timings improve or stay stable.

### 32.0 - Rhyme Section Completion Report

Default to the best proven V2 path, preserve fallback where needed, and publish a final report comparing research goals to implemented work.

Acceptance:
- Rhymes are local/offline, fast, active-line anchored, and hip-hop tuned.
- Every unimplemented research item has a documented blocker.
- Final report lists sources, shipped versions, validation, and remaining risk.
